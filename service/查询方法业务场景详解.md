# LightRAG 查询方法业务场景详解

> 本文档面向反诈知识图谱系统的开发者和使用者，详细说明每个查询方法的设计意图、适用场景和使用实例。

## 一、架构概览

系统采用 **三存储层 + 多查询路径** 的架构：

```
                    用户查询
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
       Neo4j 图     Qdrant 向量   LLM 生成
    (实体类型/关系)  (语义相似度)   (自然语言回答)
```

- **Neo4j**：管理实体节点（7种类型）和有向关系边（9种类型），负责图谱结构
- **Qdrant**：存储实体向量和chunk向量，负责语义相似度匹配
- **LLM**：基于检索到的知识生成自然语言回答

**预定义的 7 种实体类型**：
| 类型 | 说明 |
|------|------|
| `summary` | 案件简要描述（每篇文书 1 个，entity_name 为 doc_id，description 为完整摘要） |
| `FraudScenario` | 诈骗场景（10 种预定义类型） |
| `FraudFeature` | 骗局特征（欺骗性表现） |
| `FraudMethod` | 诈骗手法（技术手段） |
| `PreventionMeasure` | 防范建议（基于案情推理生成） |
| `LawRegulation` | 法律法规 |
| `RelatedCase` | 关联案例（详细案例总结，用于相似案例匹配） |

**9 种关系类型**：
| 关系 | 连接 | 说明 |
|------|------|------|
| `involves` | summary → FraudScenario | 案件涉及某诈骗场景 |
| `describes` | summary → FraudFeature | 摘要描述骗局特征 |
| `mentions` | summary → FraudMethod | 摘要提及诈骗手法 |
| `summarizes` | RelatedCase → FraudScenario | 案例涉及某诈骗场景 |
| `has_feature` | RelatedCase → FraudFeature | 案例具有某骗局特征 |
| `uses` | RelatedCase → FraudMethod | 案例使用某诈骗手法 |
| `violates` | FraudMethod → LawRegulation | 手法违反法律法规 |
| `prevents` | PreventionMeasure → FraudMethod | 防范措施预防某手法 |
| `counters` | PreventionMeasure → FraudFeature | 防范措施应对某特征 |

## 二、RAG 问答类

### 1. `query(question, mode)` — 知识图谱问答

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j 图谱 + Qdrant chunks + LLM |
| **场景** | 用户以自然语言提问，系统检索知识图谱中的相关信息后由 LLM 生成回答 |
| **参数** | `question`（问题文本）、`mode`（检索模式）、`response_type`（回答格式） |
| **返回值** | 自然语言回答字符串 |

**检索模式选择**：

| 模式 | 原理 | 适用场景 |
|------|------|---------|
| `mix`（推荐） | KG 图谱 + 向量混合检索 | 大多数问答场景 |
| `local` | 基于局部实体上下文 | 查询具体实体的详细信息 |
| `global` | 全局知识检索 | 跨实体的趋势分析 |
| `hybrid` | local + global 合并 | 需要最全面的知识覆盖 |
| `naive` | 纯向量搜索 | 不使用知识图谱的简单语义搜索 |

**实例**：

```python
# 例1：案件定性分析
answer = svc.query("这起案件属于什么类型的诈骗？", mode="mix")
# 回答基于知识图谱检索到的实体和原文 chunk 生成

# 例2：诈骗手法识别
answer = svc.query("诈骗分子使用了哪些技术手段？", mode="mix")

# 例3：法律适用分析
answer = svc.query("该案违反了什么法律条款？", mode="mix")
```

> **注意**：`query()` 的回答由 LLM 生成，内容具有不确定性。若需要确定性的结构化数据，应使用 `query_data()` 或其他具体查询方法。

### 2. `query_data(question, mode)` — 纯数据检索

| 项目 | 说明 |
|------|------|
| **数据源** | 同 query，但不经过 LLM |
| **场景** | 需要拿到结构化的检索结果自行处理，不需要 LLM 生成回答 |
| **参数** | `question`（查询文本）、`mode`（检索模式） |
| **返回值** | 字典，包含 entities / relationships / chunks / references |

**实例**：

```python
# 前端需要自己渲染"相关实体卡片"
data = svc.query_data("冒充公检法")
# 返回:
# {
#   "entities": [{"entity_name": "冒充公检法工作人员拨打电话", "entity_type": "FraudMethod", ...}],
#   "relationships": [{"source": "...", "target": "...", "relation_type": "involves"}],
#   "chunks": [{"reference_id": "1", "content": "被告人谷某敏冒充济南市公安局民警..."}],
# }
```

## 三、按实体类型查询

### 3. `query_by_entity_type(type)` — 按类型列出实体

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j 图数据库（`get_all_nodes()`） |
| **场景** | 用户想浏览或筛选某类实体 |
| **参数** | `entity_type`（实体类型名）、`top_k`（返回数量上限） |
| **返回值** | 实体信息列表，每项包含 entity_name、description、entity_type |

**实例**：

```python
# 查看所有预定义的诈骗手法
methods = svc.query_by_entity_type("FraudMethod", top_k=50)
# [{"entity_name": "冒充公检法工作人员", "entity_type": "FraudMethod", "description": "..."}, ...]

# 查看所有法律法规
laws = svc.query_by_entity_type("LawRegulation", top_k=50)
```

### 4. `query_case_summary(doc_id)` — 获取案件摘要

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j 图数据库（entity_type="summary" 的节点） |
| **场景** | 快速查看案件基本信息（被告人、手法、金额、判决结果） |
| **参数** | `doc_id`（可选，若指定则仅返回该文档的摘要） |
| **返回值** | summary 实体列表，每项包含 entity_name（doc_id）、description（完整摘要）、source_id |

**设计说明**：
- `entity_name` 使用文档 ID（doc_id），如 `"test_doc_001"` 或 `"冒充公检法类诈骗/冒充公检法类诈骗-案例一"`
- `description` 字段存储完整的案件摘要文本，格式为：`"{被告人身份+姓名}以{诈骗方式/理由}，通过{具体手段}骗取{被害人}共计{金额}，被判处{刑罚+罚金}"`
- 每个文书仅包含 1 个 summary 实体

**实例**：

```python
# 查看所有案件摘要
summaries = svc.query_case_summary()
# [{"entity_name": "test_doc_001",
#   "description": "被告人谷某敏以冒充公检法机关工作人员，谎称被害人涉嫌洗钱犯罪为由...被判处有期徒刑十二年并处罚金二十万元",
#   "entity_type": "summary",
#   "source_id": "test_doc_001"}]

# 查看特定文档的摘要
summary = svc.query_case_summary(doc_id="test_doc_001")
```

## 四、混合检索类

### 5. `query_similar_cases(question)` — 类似案件检索

| 项目 | 说明 |
|------|------|
| **数据源** | Qdrant chunks_vdb（dense + BM25） + Neo4j（summary / RelatedCase） |
| **场景** | 输入案件特征描述，查找案情相似的案例 |
| **参数** | `question`（查询描述）、`top_k`（返回数量上限） |
| **返回值** | 字典，包含 chunks / summaries / related_cases / count |

**检索流程**：
```
输入 "虚假投资理财诈骗"
    ↓
BM25 在 chunks 中找相似内容（dense 向量语义 + BM25 关键词，RRF 融合）
    ↓
从 chunk 的 full_doc_id 提取文档标识
    ↓
用关键词匹配 summary 和 RelatedCase 实体
    ↓
返回匹配的案件摘要、关联案例及涉及的文档
```

**实例**：

```python
cases = svc.query_similar_cases("虚假投资理财诈骗", top_k=5)
# {
#   "chunks": [{"content": "...", "rrf_score": 0.016}],
#   "summaries": [{"entity_name": "test_doc_001", "description": "..."}],
#   "related_cases": [{"entity_name": "张某虚假网络投资理财诈骗案", "description": "..."}],
#   "query": "虚假投资理财诈骗",
#   "count": 2
# }
```

### 6. `query_prevention(query)` — 防范措施查询

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j（PreventionMeasure 节点） + entities_vdb（语义搜索排序） |
| **场景** | 查询防范建议，用于反诈宣传、知识库展示 |
| **参数** | `query`（可选，查询文本；若为 None 则返回所有防范措施） |
| **返回值** | 防范措施列表，每项包含 entity_name、description |

**检索策略**：先尝试用语义搜索排序，若无匹配结果则 fallback 到双字 bigram 关键词模糊匹配。

**实例**：

```python
# 无查询条件 → 返回所有防范措施
items = svc.query_prevention()
# [{"entity_name": "核实平台资质", "description": "投资前应核实交易平台是否具备合法资质..."}, ...]

# 带查询条件 → 语义搜索 + 关键词匹配 fallback
items = svc.query_prevention(query="虚假投资")
# [{"entity_name": "核实政府项目真实性", "description": "..."},
#  {"entity_name": "不轻信非正规渠道的投资机会", "description": "..."}, ...]
```

### 7. `query_with_bm25(question)` — Dense + BM25 混合检索

| 项目 | 说明 |
|------|------|
| **数据源** | Qdrant chunks_vdb |
| **场景** | 需要同时匹配语义相似度和关键词精确匹配的原始 chunk |
| **参数** | `question`、`top_k`、`dense_weight`（默认0.5）、`bm25_weight`（默认0.5） |
| **返回值** | 包含 chunks 列表的字典，每项包含 content 和 rrf_score |

**实例**：

```python
results = svc.query_with_bm25("冒充公检法", top_k=5)
# dense 向量匹配到: "被告人谷某敏冒充济南市公安局民警"（语义相似）
# BM25 关键词匹配到: "冒充公检法机关工作人员"（精确匹配）
# 通过 RRF 融合排序，两者都命中的结果排最前
# 即使 BM25 sparse 索引不可用（降级为 dense-only），也会计算 rrf_score
```

## 六、图谱分析类

### 8. `get_entity_centrality(type)` — 实体中心性排名

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j 图数据库（所有节点和边的度数计算） |
| **场景** | 发现最核心/最常见的实体，用于发现高频诈骗手法、关键法律条款等 |
| **参数** | `entity_type`（可选过滤）、`top_k`（排名上限） |
| **返回值** | 实体列表，按 degree（连接数）降序排列 |

**实例**：

```python
top_methods = svc.get_entity_centrality("FraudMethod", top_k=10)
# [{"entity_name": "冒充公检法工作人员", "degree": 12},
#  {"entity_name": "伪造交易盈利截图", "degree": 8}, ...]
```

### 9. `get_shortest_path(src, tgt)` — 实体间最短路径

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j shortestPath Cypher 查询 |
| **场景** | 分析两个实体之间的关联链路 |
| **参数** | `src_entity`、`tgt_entity`、`max_depth`（最大深度，默认4） |
| **返回值** | 路径信息字典（nodes + edges），若无路径则返回 None |

**实例**：

```python
path = svc.get_shortest_path("伪造交易盈利截图", "核实平台资质")
# {
#   "nodes": [{"entity_name": "伪造交易盈利截图", ...}, {"entity_name": "核实平台资质", ...}],
#   "edges": [{"relation_type": "uses", ...}, {"relation_type": "prevents", ...}]
# }
```

### 10. `get_entity_neighbors(entity, depth)` — N 跳邻居

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j Cypher 路径查询 |
| **场景** | 查看某实体的直接关联实体（1跳）或更深层次的关联网络（2跳+） |
| **参数** | `entity_name`、`depth`（搜索深度，默认1） |
| **返回值** | 包含 center、neighbors、edges 的字典 |

**实例**：

```python
nb = svc.get_entity_neighbors("冒充公检法类诈骗", depth=1)
# {
#   "center": "冒充公检法类诈骗",
#   "neighbors": [
#     {"entity_name": "冒充公检法工作人员", "entity_type": "FraudMethod"},
#     {"entity_name": "刑法第二百六十六条", "entity_type": "LawRegulation"},
#     ...
#   ],
#   "edges": [
#     {"source": "冒充公检法类诈骗", "target": "冒充公检法工作人员", "relation_type": "mentions"},
#     ...
#   ]
# }
```

## 七、基础查询

### 11. `get_entity_info(name)` — 实体详情

| 项目 | 说明 |
|------|------|
| **数据源** | Neo4j 图数据库 + Qdrant entities_vdb |
| **场景** | 查看单个实体的完整信息 |
| **返回值** | 包含 entity_name、description、entity_type、source_id、graph_data（含属性） |

**实例**：

```python
info = svc.get_entity_info("冒充公检法工作人员")
# {
#   "entity_name": "冒充公检法工作人员",
#   "entity_type": "FraudFeature",
#   "description": "诈骗分子虚构自己为公安、检察或法院工作人员身份以获取被害人信任",
#   "graph_data": {"attr_出现频次": "多次", "attr_欺骗性表现": "利用国家机关权威制造可信度", ...}
# }
```

### 12. `get_relation_info(src, tgt)` — 关系详情

**实例**：

```python
info = svc.get_relation_info("谷某敏冒充公检法类诈骗案", "冒充公检法类诈骗")
# 返回关系类型、evidence、weight、关联文本块等
```

### 13. `get_graph_labels()` — 获取所有实体名称

| 项目 | 说明 |
|------|------|
| **场景** | 获取知识图谱中所有实体的名称列表（注意：是实体名而非实体类型） |
| **返回值** | 实体名称字符串列表 |

> **注意**：该方法返回的是图谱中每个实体的 `entity_name`（如"冒充公检法类诈骗"、"刑法第266条"），而非实体类型分类。如需按类型统计，应使用 `query_by_entity_type()`。

### 14. `get_knowledge_graph(entity_name, max_depth, max_nodes)` — 知识图谱子图

| 项目 | 说明 |
|------|------|
| **场景** | 获取指定实体周围的子图，用于可视化展示 |
| **返回值** | `KnowledgeGraph` 对象，包含 nodes 和 edges |

**注意事项**：
- Neo4j 中节点的 `id` 是内部数字标识（如 0, 1, 27...），实体的实际名称在 `properties.entity_name` 中
- 边的 `source`/`target` 也是数字 ID，需通过节点映射转换为可读的实体名

**实例**：

```python
# 获取全图谱
graph = svc.get_knowledge_graph("*", max_depth=1, max_nodes=1000)
print(f"节点数: {len(graph.nodes)}, 边数: {len(graph.edges)}")

# 获取某实体的子图
graph = svc.get_knowledge_graph("冒充公检法类诈骗", max_depth=2, max_nodes=100)
```

### 15. `get_processed_doc_ids()` — 已处理文档列表

| 项目 | 说明 |
|------|------|
| **数据源** | MongoDB doc_status（DocStatus.PROCESSED） |
| **返回值** | 已处理文档的 ID 列表 |

## 九、典型业务流程场景

### 场景1：案件分析

```
用户上传一篇裁判文书 → direct_ingest 处理 → 生成 18 个实体 + 13 条关系

用户提问："这起案件属于什么类型的诈骗？"
  → query("这起案件属于什么类型的诈骗？")
  → 检索到 FraudScenario 实体"冒充公检法类诈骗"
  → LLM 生成回答
```

### 场景2：反诈知识库构建

```
批量导入 100 篇裁判文书 → 知识图谱自动构建

管理员查看系统有哪些诈骗手法：
  → query_fraud_methods() → 列出所有 FraudMethod 实体

管理员查看系统有哪些防范建议：
  → query_prevention() → 列出所有 PreventionMeasure 实体
```

### 场景3：类似案件匹配

```
用户输入"刷单返利诈骗"
  → query_similar_cases("刷单返利诈骗")
  → BM25 在 chunks 中匹配到 3 篇相关文书
  → 返回这 3 篇文书的 summary（doc_id + 摘要文本）和 RelatedCase
  → 用户可以看到类似案情的案件详情
```

### 场景4：诈骗手法关联分析

```
风控人员想了解"伪造交易截图"和"防范建议"之间的关联
  → get_shortest_path("伪造交易盈利截图", "核实平台资质")
  → 返回关联路径：手法 → 特征 → 防范措施
  → 帮助理解哪些防范措施针对哪些诈骗手法
```

### 场景5：反诈宣传素材收集

```
宣传人员需要收集"针对投资理财诈骗的防范建议"
  → query_prevention(query="投资理财")
  → entities_vdb 语义搜索匹配到相关防范措施（无匹配时 fallback 到关键词）
  → 返回"核实平台资质"、"警惕高收益承诺"、"不轻信陌生来电"等
  → 用于制作反诈宣传材料
```
